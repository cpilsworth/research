Adobe Dynamic Media Video Player Web Component

This experiment wraps Adobe's Dynamic Media HTML5 video player in a reusable Web Component so it can be embedded directly in another page without relying on an iframe.

Custom element: <adobe-dm-video-player> Uses Adobe hosted viewer runtime Designed for direct page embedding

What This Folder Contains

  • adobe-dm-video-player.js: the custom element implementation.
  • index.html: the demo page for this published site.
  • serve-local.js: the tiny static server used for local testing in the repo.

Usage

<script type="module" src="/path/to/adobe-dm-video-player.js"></script>

<adobe-dm-video-player
  src="https://delivery-p31359-e1338271.adobeaemcloud.com/adobe/assets/urn:aaid:aem:7091fd88-9d0f-4aa6-8bfb-2746ce1d815f/play"
  controls
>
</adobe-dm-video-player>

Supported Attributes

  • src: required Dynamic Media /play URL.
  • token: optional secure-delivery token.
  • autoplay: optional boolean attribute.
  • controls: optional boolean attribute, defaults to true.
  • letterboxed: optional boolean attribute, defaults to false.
  • proxy-base-path: optional same-origin proxy base path when a host server rewrites manifests and media.

How It Works

The component loads Adobe's hosted viewer CSS and JavaScript once per page, derives the HLS manifest, DASH manifest, and poster image from the supplied /play URL, and mounts the Adobe player into a normal DOM container.

It uses light DOM rather than shadow DOM because Adobe's viewer runtime looks up its mount node with document.getElementById(...).

Special Video Processing

Dynamic Media manifests can contain relative media references. When a host page uses a different origin, those references may need to be rewritten through a same-origin proxy path so the player can still resolve child playlists, MP4 fragments, and DASH assets correctly.

DASH rewrites also need valid XML escaping when query parameters are inserted into MPD attributes. Without that, the browser can reject the source as unsupported even when the media itself is valid.

Benefits And Tradeoffs

Compared with an iframe, the Web Component integrates much more cleanly with the host page: layout is simpler, responsiveness is better, and host code can interact with the player more naturally.

The tradeoff is that the embedding page now owns more complexity. Script loading, player lifecycle, cross-origin media handling, and any manifest proxying are the responsibility of the host implementation instead of Adobe's standalone player page.
